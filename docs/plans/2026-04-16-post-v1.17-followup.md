## P1 — Bugs / correctness

### Task 1: Fix `understand.focus` filter (bug)

**Symptom**: `understand(symbol="toggle", focus="ThemeToggle.svelte")` returns `symbol "toggle" not found` even though `symbol_search(query="toggle")` finds it in that exact file (line 8).

**Location**: `internal/compound/understand.go` (or wherever `understand` resolves focus). Search for `focus` usage:
```
grep -rn "focus" internal/compound/ cmd/go-code/tool_understand.go
```

**Investigation**:
- Read how focus is matched against symbol.File path. Likely a full-path match vs substring match mismatch. File paths come as `/host/src/piter-now/frontend/src/components/ThemeToggle.svelte` while user passes bare filename.
- Probably should treat focus as substring of file path OR try suffix match OR try glob.

**Acceptance**:
- `understand(symbol="toggle", focus="ThemeToggle.svelte")` returns exactly the ThemeToggle match.
- `focus="components/Filters"` returns symbols in `frontend/src/components/Filters.svelte`.
- Add 2 table-driven tests for the focus matching.

**Update (2026-04-16)**: shipped in `e370046`. Root cause was NOT in `internal/compound/understand.go`: the `Focus` value was passed into `callgraph.BuildFromRepo` → `ingest.IngestRepo`, which uses prefix/glob matching on relative paths and excluded the matching files BEFORE any compound-level filtering ran. Fix: stop passing `Focus` to `BuildFromRepo` for `understand` (other tools — `dead_code/impact/prepare_change/dataflow_dead/code_research/explore` — keep ingest-level prefix scoping for performance), add unexported `filterByFocus` helper in `cmd/go-code/tool_understand.go` with 3-tier match (exact → suffix → substring). Tests in `cmd/go-code/tool_understand_test.go` including `TestFilterByFocus_IngestLayeringRegression` documenting the layering invariant.

### Task 2: Test that preproc handles attribute `>` bound correctly

**Symptom**: During code review of Task 2 we noted the fix in `internal/parser/preproc/svelte.go` and `astro.go` bounds the `>` search to 512 bytes / one line. This was introduced but not exercised by a test.

**Location**: `internal/parser/preproc/svelte_test.go`, `astro_test.go`.

**Acceptance**: add a fixture with `<script lang="ts" src="x&gt;y.js">content</script>` (attribute value literally containing `>`). Verify ExtractSvelte picks up the real tag close, not the attribute-internal `>`.

**Update (2026-04-16)**: the `&gt;` fixture above does NOT actually exercise the bound — `&gt;` is 4 bytes (`&`,`g`,`t`,`;`) with no `>` byte, so `bytes.IndexByte('>')` skips it regardless of any cap. Tests pass for a coincidental reason. Real bound coverage requires fixtures where `>` lands beyond the bound: (a) `\n` immediately after `<script` so the real `>` is on the next line (single-line cap), and (b) ≥513 bytes of attribute padding so `>` sits past the byte cap. Shipped in `e08a391` + `07a7bc9`: `BoundSingleLine` + `BoundMaxBytes` for both Svelte and Astro, plus literal `512` lifted to package const `tagOpenScanLimit` (preproc.go) so tests reference `tagOpenScanLimit+88` and break compile-fast on const removal. `AttrLiteralGt` tests (Option A) pin current limited behavior of literal-`>` attribute parsing; will fail loudly if escape handling lands.
